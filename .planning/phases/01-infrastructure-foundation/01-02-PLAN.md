<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Host → containers | Apenas Django (3600) e Postgres (3601) expostos; Redis/ChromaDB/Ollama só na rede interna |
| Processo do container → SO host | Container roda como duser (não-root) — limita escalonamento em caso de comprometimento |
| Rede Docker interna ↔ rede externa | Redis e serviços de suporte sem porta no host eliminam vetor de ataque externo |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-05 | Elevation of Privilege | Container Django/Celery | mitigate | `USER duser` no Dockerfile — container nunca roda como root (verificável: grep `USER duser`) |
| T-01-06 | Information Disclosure / Tampering | Redis (broker) | mitigate | Redis sem `ports:` no docker-compose (D-08) — inacessível do host e da rede externa |
| T-01-07 | Denial of Service | Dependência supply-chain (uv sync) | mitigate | `uv sync --frozen` + `uv.lock` commitado — versões fixas, sem upgrade silencioso em build |
| T-01-08 | Tampering | ChromaDB image tag | mitigate | Tag semântico estável (0.6.x) em vez de `:latest` — evita pull de build de desenvolvimento instável (Pitfall 6) |
| T-01-09 | Spoofing / Information Disclosure | Headers HTTP, HSTS, X-Frame-Options | transfer | Hardening de headers, rate limiting e SAST são responsabilidade explícita da Phase 7 (D-10) |
| T-01-10 | Information Disclosure | SECRET_KEY ausente em runtime | mitigate | entrypoint.sh valida `[ -z "$SECRET_KEY" ]` e aborta com exit 1 antes de qualquer comando Django |
</threat_model>
